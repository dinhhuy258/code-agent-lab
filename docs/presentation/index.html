Foundation

What is a Coding Agent?

A coding agent wraps an LLM in a control system that can observe, reason, act, and iterate autonomously.

💬 Chatbot

Single turn. You ask, it answers. No memory, no tools, no persistence. Just raw text completion.

Stateless

✨ Copilot

Suggests code inline as you type. Context-aware within the current file, but no autonomy or tool use.

Reactive

🤖 Coding Agent

Autonomous loop: reads files, executes commands, writes code, validates results. Multi-step reasoning with tool access.

Autonomous
💡
Key insight: Much of what appears to be “model quality” is actually context quality and surrounding system design. The harness matters as much as the model.
Motivation

Why Build One?

Understanding agent internals transforms how you use and build AI-powered tools.

The Problem Space

🔍 Raw LLMs are stateless — they don't know your codebase, can't run tests, can't read files

🔄 Multi-step tasks require loops — read, analyze, modify, verify, repeat

🛡️ Safety requires guardrails — confirmation for writes, isolation for sub-tasks

Core Architecture

The ReAct Loop

Reasoning + Acting — the agent iterates between thinking and doing until the task is complete.

STEP 1
📥
Observe
Receive user message, collect environment context
→
STEP 2
🧠
Reason
Send full history + tools to LLM, get structured response
→
STEP 3
🛠️
Act
Execute tool calls, append results to history
→
STEP 4
🔁
Loop
Repeat until text-only response or max turns (25)
💡
The model controls the loop. The harness is the executor; the LLM decides when to call tools, which tools to call, and when the task is done.
Implementation

AgentClient.send()

The core loop from our codebase — surprisingly simple at its heart.

core/agent_client.py
async def send(self, user_text: str): self._session.append_user_message(user_text) for _ in range(MAX_TURNS): # Send history + tools to LLM result = await self._session .send_message_stream() # Yield text chunks for streaming for chunk in result.text_chunks: yield TextChunk(chunk) # No tool calls? We're done if not result.function_calls: yield TextResponse(result.text) return # Execute tools, loop back await self._process_tool_calls( result.function_calls )

🔄 The Loop

Up to 25 iterations. Each iteration: call LLM → check for tool calls → execute or return.

📝 History is Memory

The conversation history is the agent's memory. Every message, tool call, and result stays in the list.

⚠️ Errors are Messages

Tool failures return as ToolResult.error — the model sees the error and can adjust its approach.

Architecture

Four-Layer Stack

Each layer has a single responsibility and can evolve independently.

Layer 4 — Orchestrator
AgentClient
The agent loop. Decides when to stop.
Layer 3 — History Manager
ChatSession
Maintains conversation history. Appends messages.
Layer 2 — Turn
Turn
Single request/response cycle. Extracts tool calls.
Layer 1 — LLM Client
GeminiLLMClient
Talks to the API. Handles auth, errors, streaming.
🏗️
Why layers? New features just "fill in" existing layers. Adding streaming didn't require rewriting the agent loop. Adding sub-agents didn't change the tool system.
Tool Use

The Fundamental Primitive

Tool use is the only primitive an LLM actually understands for interacting with the world.

Define Schema
Declare tools with JSON Schema — name, description, parameters
→
2
📤
Send to API
Tool declarations sent alongside conversation history
→
3
🎯
Parse Calls
Model returns structured function_call objects (not free text)
→
4
📥
Return Result
Execute locally, send result back as function_response
⚠️
The LLM never executes code. It outputs JSON describing what tool to call. Your code (the harness) is the executor.
Implementation

Tool Abstraction

Every tool implements the same interface. The registry dispatches by name.

tools/base.py
class BaseTool(ABC): def get_name() -> str: # Unique tool identifier ... def get_declaration() -> ToolDeclaration: # JSON Schema for the LLM ... def execute(**kwargs) -> ToolResult: # Do the thing ... def needs_confirmation() -> bool: # Safety flag return False
tools/registry.py
class ToolRegistry: def get_declarations(): # Collect all schemas for API return [t.get_declaration() for t in self._tools] def execute(name, **kwargs): # Dispatch to the right tool tool = self._tools[name] return tool.execute(**kwargs)
🔌
Pluggable: New tools just register into the same dictionary. The agent loop and LLM don't care how many tools exist.

Real-Time UX

Streaming Architecture

Without streaming, users stare at a blank screen. The event system solves this.

❌ Before: Blocking

User sends message → 5-30 seconds of nothing → full response appears at once. No feedback, no progress.

✅ After: Streaming Events

Tokens appear as they're generated. Tool calls show names and progress. Sub-agent activity visible in real-time.

core/events.py
# Event types the UI subscribes to TextChunk # streaming token TextResponse # final complete text ToolCallStart # tool name + args ToolCallUpdate # sub-agent progress ToolCallEnd # success or error UsageUpdate # token metrics
Implementation

Event Dispatch Pattern

Background thread yields events; UI thread renders them via call_from_thread.

app.py
# Background thread @work(thread=True) def _run_llm(self, user_text): for event in agent.send(user_text): if isinstance(event, TextChunk): self.call_from_thread( _on_text_chunk, event ) elif isinstance(event, ToolCallStart): self.call_from_thread( _on_tool_call_start, event ) elif isinstance(event, ToolCallEnd): self.call_from_thread( _on_tool_call_end, event )

🎨 Textual TUI

Uses the Textual framework for rich terminal UI. Widgets for chat messages, tool calls, status bar, and thinking indicators.

🧵 Thread Safety

call_from_thread() bridges the background worker thread to the main UI thread safely. Events are posted, not pushed.

📡 Progressive Rendering

Each TextChunk accumulates text in the widget. Textual diffs efficiently — only changed parts re-render.

The API is stateless. Every turn resends the entire conversation history. Cost grows quadratically.

📈 Quadratic Cost

Total tokens ≈ N × fixed + N(N+1)/2 × delta. 30 turns = 56x the cost of turn 1.

💨 Context Pollution

1 file read = 5K tokens permanently stuck in history. 10 reads = 50K resent every turn.

Implementation

The SKILL.md Format

YAML frontmatter for metadata, Markdown body for full instructions.

.code-agent/skills/hello/SKILL.md
--- name: hello description: Triggers when the user says hello --- # Hello Skill 1. When activated, greet the user warmly 2. Locate the full path of the `debug_client.py` file 3. Read and summarize its contents
📁
Discovery: Skills are auto-discovered from .code-agent/skills/ at startup. No registration needed.

🏷️ Frontmatter (metadata)

name + description only. This goes into the system prompt. Fallback: directory name if missing.

📝 Body (instructions)

Full detailed instructions. Only injected into conversation when activate_skill is called.

⚙️ Wiring

SkillManager discovers → ActivateSkillTool registered if skills exist → zero overhead when unused.

Sub-Agents

Context Isolation

A single conversation with one system prompt can't handle everything. Sub-agents solve three problems.

💥 Context Pollution

Intermediate tool results (file reads, greps) stay in history forever, bloating every subsequent API call.

🎭 No Specialization

One system prompt can't optimize for all personas. A code reviewer needs different instructions than a debugger.

⏳ Sequential Only

With a single ReAct loop, tasks execute one at a time. No parallel investigation possible.

💡
The solution: A sub-agent is a tool that runs its own ReAct loop in an isolated ChatSession with its own system prompt. Results return to the parent; intermediate work is discarded.

Data Flow

History Isolation

The sub-agent's entire history is created, used, and garbage-collected.

Main conversation history
[user] "find auth validation" [model] call task( prompt="Search for token validation" ) [user] tool_result: "Task Finished: Tokens validated at auth.py:42" [model] "Found it at auth.py:42"
✨
Main context stays clean. Only the summary comes back — not the 5 file reads and 3 greps the sub-agent did.
Sub-agent history (discarded)
[user] "Search for token validation" [model] call grep_search(...) [user] function_response: "auth.py:42" [model] call read_file(...) [user] function_response: "def validate..." [model] "Tokens validated in src/auth.py:42 via validate_token()" ^ entire history discarded

📊 Impact

Same work in main context: +70K tokens. Via sub-agent: 35K main + isolated 20K. Context shrinks 52%.

🏗️ Adapter Pattern

The McpTool wrapper makes MCP tools indistinguishable from local tools. The ToolRegistry, agent loop, and LLM don't know the difference.

Takeaways

What We Learned

Building a coding agent from scratch reveals truths you can't learn from using one.

🎯
Context is the Bottleneck
#1 insight

Not raw intelligence. The quality of context you provide — and how efficiently you manage it — determines agent effectiveness.

🔀
Routing, Not Intelligence
architecture

"Agentic AI is a routing problem, not an intelligence problem." Getting the right context to the right place at the right time.

📦
Lazy Loading is Essential
optimization

Deferred tools (name only at startup), deferred skills (metadata only), sub-agent isolation. Never load what you don't need.

🛡️
Safety is First-Class
design

Confirmation flags on dangerous tools, isolated sessions for sub-agents, max turn limits. Guardrails are architecture, not afterthoughts.

🧱
Never Rewrite on Feature Add
engineering

Layered architecture means new chapters just "fill in" existing layers. Streaming didn't rewrite the loop. Sub-agents didn't change tools.

💬
Errors are Messages
pattern

Tool failures return as function_response, not exceptions. The model sees errors and adjusts. Self-healing through context.
